Reconciliation, the receivable, & the Phase 8 break

An interactive walk-through of the three reconcile checks — and why only one of them can be fixed by the agent posting an entry.

1 · The big picture: three records of the same money

Every rupee shows up in three independent places. Reconciliation proves they agree.

RecordFile / sourceWhat it claims
Settlements rawrazorpay/settlements.json"I batched these payments & refunds and deposited ₹X to your bank."
Bank feed rawbank-feed.json"I (the bank) credited ₹Y on this date." (independent)
The books ledgerposted by closerThe double-entry GL the close produces from events.

Pipeline that builds the books:

refunds.json→ ingest.Normalize→ events→ classify→ ledger.Post→ the books
The single most important fact: two of the three checks read raw files only. Just one — check #3 — reads the ledger. The agent can only change the ledger. So the agent can only ever fix check #3. Hold onto that; everything below leads there.

2 · One example batch (used everywhere below)

ItemAmount (₹)Note
Payment pay_A1,000a sale
Payment pay_B1,000a sale
Refund rfnd_X1,000full refund of pay_B
Razorpay fees40on the batch

The seeder computes the actual bank deposit by netting it all out (generate.go):

net_deposit = Σpay − Σrefund − Σfees = 2000 − 1000 − 40 = 960 ← this number is stored on the settlement

So on disk the settlement says: net_deposit = 960, payment_ids = [pay_A, pay_B], refund_ids = [rfnd_X], UTR = "UTR_abc".

3 · The three checks

Check #1 — Settlement → Bank  reads raw only

"Did the cash actually arrive?" Match each settlement to a bank credit by UTR, then compare amount & date.

settlement setl_1: net_deposit = 960 UTR = "UTR_abc" date = May 10 bank feed: credit = 960 ref = "UTR_abc" date = May 11 match on UTR ✅ amount 960 == 960 ✅ |May10 − May11| = 1d ≤ 3d ✅ → no break

Fires if: no matching credit, wrong amount, or the date is outside tolerance. That's a real cash discrepancy — you escalate it; no posting can conjure missing cash.

Check #2 — Batch-sum  reads raw only

"Does the receipt add up?" Ignore the bank entirely. Take the settlement's own listed members and re-derive the deposit. Does it equal the printed total?

The inject (injectRefundInBatch) only erases rfnd_X from the list — it never touches net_deposit (still 960). It's like erasing one line item from an itemized receipt but leaving the printed total: now the line items sum to a different number, off by exactly the erased line (the ₹1,000 refund).

Fires if the member list doesn't reconstruct the deposit. That's a raw-data inconsistency — you fix it by correcting the batch list, not by posting to the ledger (posting can't change settlements.json).

Check #3 — Receivable-clears  reads the ledger

"Did the money in flight land?" The assets/razorpay-settlement-receivable account is a temporary bucket — money customers paid that Razorpay hasn't deposited yet. It should drain to ~0 by period end.

4 · Why only check #3 is fixable by the agent

The agent's only lever is returning {entry_type, params} that the Go ledger validates and posts (SPEC §8). Posting changes the ledger. Only check #3 reads the ledger. Therefore:

CheckThe issue it raisesPosting fixes it?
#1 settlement→bankcash missing / wrong amount / wrong dateNo → escalate (real cash gap)
#2 batch-summember list doesn't sum to depositNo → correct raw batch data
#3 receivable-clearssomething settled-but-not-bookedYes → book the missing entry
The chain that makes Phase 8 work:
investigate books the missing refund_reversal → posts Cr receivable −1000 → receivable: +1000 − 1000 = 0 → re-run check #3: 0 == 0 ✅ break clears → that refund_reversal now matches truth ✅ score rises to 100%

5 · The Phase 8 decision (what I'm asking you)

Today's --inject refund-in-batch produces a check #2 break, and the refund stays booked (score already 100%). So the handover's goal — "investigate adds a posting → reconcile passes → score rises" — is structurally impossible with that break (check #2 can't be cleared by posting, and re-booking an already-booked refund would double-count).

Option 1 — recommended

Seed a refund the deterministic close can't book (incomplete tax metadata — the §1/§2 long-tail). It surfaces as a check #3 residual. Investigate recovers the rate from orders.json + the id/amount from refunds.json, books the refund_reversal.

→ receivable clears, reconcile passes, score 100%. Faithful to SPEC §7#3 "settled-but-not-booked" and the §8 "postings to add" contract. Single clean break.

Option 2 — literal handover

Keep today's check #2 break (refund stays booked). Investigate returns an empty postings list + a batch correction; reconcile passes on re-check.

→ no posting added, score stays 100% (no rise).

Option 3 — both

Seed both a check #3 (posting-resolvable) and a check #2 (data-correction) break to show both paths. More fixtures, more closer complexity.
