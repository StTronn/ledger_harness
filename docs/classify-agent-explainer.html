Why the miss triggers the agent

The dtc_sale books a GST-inclusive gross split into net revenue + output GST. To split it, the rule needs the rate:

net = gross × 100/(100+rate) gst = gross − net with rate=5: net = 386158×100/105 = 367769 gst = 18389 ✓ (367769+18389 = 386158) with rate="": cannot divide — no rate → the rule REFUSES (returns ok=false, "missing gst_rate")

That ok=false is the only trigger. The rule never guesses a rate. The orchestrator then asks the agent, which fetches the order, recovers rate=5, and returns the SAME {dtc_sale, params} the rule would have produced — so the booked entry is byte-identical to truth.
